# Notes

## 2026-04-21

- `flutter_local_notifications` v21 switched `show()` and `cancel()` from positional to named parameters. The existing `cancelReminder(id: id)` call was already correct (coincidentally); new notification methods must use `show(id:, title:, body:, notificationDetails:)` and `cancel(id:)`.
- `FocusModeNotifier.startFocus` reads `databaseProvider` and `currentUserIdProvider` eagerly before the `await` to avoid the autoDispose-ref-after-await bug (same pattern established in `TaskDetailNotifier`).
- `inProgress → deferred` (Abandon) clears `selectedForToday` to false via `_buildTransitionCompanion`, so the task is immediately removed from today's focus list and returned to Next Actions.

## 2026-04-20

- Login/register screens relied on GoRouter's `/login`→`/inbox` redirect for post-auth navigation, but when the screen is pushed on top of another route (e.g. Settings → Sign in to sync) the redirect doesn't pop the pushed entry and `_isLoading` never resets — spinner gets stuck. Fix: explicitly `context.pop()` when `canPop`, fall back to `setState(_isLoading = false)` otherwise and let the redirect take over.
- Sign-out reset `currentUserIdProvider` to `'local'` without reassigning the user's rows, so queries (all filtered by `user_id`) returned nothing and the UI promise "Your local data will remain on this device" was a lie. Fix: mirror the login-time migration — before resetting, move todos/tags/todo_tags from the real user id back to `'local'`. Round-trips cleanly through a logout/login cycle via the existing `_handleMigration`.
- First login fix wasn't enough when a conflict-resolution dialog fires: GoRouter's `/login`→`/inbox` redirect (triggered by `authStateNotifier` flipping during login) raced with the explicit `context.pop()`, and users landed back on login. Solution: drop the redirect entirely — login/register screens own their post-auth navigation (`pop` if pushed, `go('/inbox')` otherwise). Also dropped the forced `context.go('/inbox')` after logout in Settings; the `ValueListenableBuilder<authStateNotifier>` already rebuilds the sync section in place.
- Even without the redirect, `authStateNotifier` in `refreshListenable` caused the same pop-back failure: flipping it triggers a GoRouter refresh that rebuilds the route stack from the current URI, which drops imperatively-pushed entries (the `/settings` below `/login` disappears, so `canPop` flips to false mid-flow). Fix: remove `authStateNotifier` from `refreshListenable`. No redirect depends on it anymore. The bug only manifests when the conflict dialog fires (local + server data both exist), which is why a fresh install appears to work — no local data, no dialog, no race.
- Session wasn't restored across app restarts: `authTokenProvider` is an `AsyncNotifierProvider` whose `build()` reads stored tokens and flips `authStateNotifier`, but Riverpod providers are lazy. Nothing at startup watched it (router reads the raw `authStateNotifier.value`; shell/inbox only watch `currentUserIdProvider`), so the token sat in secure storage until the user opened Login/Settings. Fix: make `JeevesApp` a `ConsumerWidget` and `ref.watch(authTokenProvider)` at the root so `build()` runs at app launch. No refresh hooks needed — `currentUserIdProvider` is set inside the same `build()`, and all data providers `watch` it.
- Restoring auth at cold start left PowerSync disconnected (tasks showed but the sync icon read "local only"). `powerSyncInstanceProvider.build()` called `await applyUser(ref.read(currentUserIdProvider))` *before* setting up `ref.listen`. On cold start the read returned `'local'` (auth still in flight), and the `'local' → realUser` flip happened in the `await`'s microtask gap, before the listener was installed. Riverpod's `ref.listen` doesn't fire for existing state, so the transition was lost and PowerSync stayed disconnected. Fix: install the listener first, then do the initial apply — the existing `pending` serial chain handles any concurrent transition correctly.
- Fix #1 wasn't enough: the sync icon still showed "local only" after restart. PowerSync's `db.statusStream` emits on **changes only**, so a subscriber that attaches after the DB has already reached a stable state (as happens on cold start, when auth/PS setup finish before `syncStatusProvider`'s async\* subscribes) waits forever — `asData` stays `null` and both the navbar and Settings fall back to the "local only" cloud icon. Fix: seed the stream with `db.currentStatus` before yielding from `statusStream`.
- PowerSync exposes `todos`, `tags`, `todo_tags` as SQLite views with INSTEAD OF triggers. SQLite forbids UPSERT (`INSERT ... ON CONFLICT DO UPDATE`) on views at the parser level — even with triggers present. Use `INSERT OR REPLACE` (`InsertMode.insertOrReplace`) instead; PowerSync's triggers handle it correctly.
- The INSERT OR REPLACE fix wasn't enough for tag assignment: `TodoTags` in Drift had no `id` column, so the emitted INSERT didn't include `id`, but PowerSync's INSTEAD OF trigger writes `NEW.id` into `ps_data__todo_tags` (which has `id` as PK). Tests passed because they use `NativeDatabase` (real table, composite PK), not a PowerSync view. Fix: add `id` to `TodoTags` and derive it deterministically from `(todo_id, tag_id)` via `uuid.v5(Namespace.url, 'jeeves://todo_tag/$todoId/$tagId')` so re-assigning the same tag collapses under INSERT OR REPLACE instead of piling up junction rows.
- The accompanying `schemaVersion` bump (5→6) tripped Drift's onUpgrade at app startup: `ALTER TABLE "todo_tags" ADD COLUMN "id" TEXT NOT NULL` throws `SqliteException(1): Cannot add a column to a view` because PowerSync exposes `todo_tags` as a view. PowerSync already manages the column set via `powersyncSchema`, so the ALTER is unnecessary on prod but still required for `NativeDatabase.memory()` tests. Fix: route every `m.addColumn` through `_addColumnIfTable`, which checks `sqlite_master.type` and skips if the target is a view. Previous migrations (from<2..from<5) were switched to the helper too; they only didn't explode earlier because they never ran on an already-installed prod DB.
- After fixing the schema-and-migration side, tag assignment still did nothing silently. Real cause: `TaskDetailNotifier` held a `Ref` and did `_ref.read(databaseProvider)` in every method. `taskDetailNotifierProvider` is `Provider.autoDispose.family`; no widget watches it, so Riverpod disposes the ref between the `ref.read()` that creates the notifier and the second `_db` access after an `await`. The callback's error was hidden by `.ignore()` in the UI. Fix: capture `db` and `userId` eagerly at construction and drop `Ref` entirely — safe because `GtdDatabase` is a process-wide singleton and the task-detail screen pops on logout.
